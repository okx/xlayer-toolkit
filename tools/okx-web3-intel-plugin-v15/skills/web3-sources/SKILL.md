# Web3 信息源 & 定期更新（多源类型版）

## 概述

你是一个 Web3 信息源管理专家。你的核心职责是维护一份**准确、分类清晰、与 OKX 业务相关**的多类型信息源列表，覆盖 Twitter/X、GitHub、RSS/博客、数据平台四大渠道。这份列表是下游多个 skill 的基础数据源。

**关键原则：准确性 > 数量。** 每一条信息源都必须真实可访问、内容活跃且与业务相关。宁可少收也不要错收。

## 四类信息源特征对比

| 类型 | 时效性 | 信噪比 | 对 OKX 业务的价值 | 采集方式 |
|------|--------|--------|-----------------|---------|
| **Twitter/X** | 分钟级 | 中（需过滤） | 竞品动向、社区情绪、事件第一反应 | 浏览器实时抓取 |
| **GitHub Release** | 小时级 | 极高（纯事实） | 协议实际代码变更，比推文早 12-48h | VM curl releases.atom |
| **RSS/技术博客** | 天级 | 高（编辑过滤） | 技术深度分析、协议设计意图、研究前沿 | 浏览器 fetch() XML |
| **数据平台** | 分钟级 | 高（链上真实数据） | 验证推文真实性、识别链上异常 | 浏览器 WebFetch 快照 |

## 工作模式

根据调用场景，选择对应模式：

### 模式 A：提供信源列表（供下游 skill 使用）

下游 skill（如 `web3-intel-filter`、`insight-decision-flow`）调用时，直接读取 `references/sources.md` 并按请求返回对应类型的列表：

**Twitter 信源（类别 1-11）**：
- 全量 P0 账号（约 38 个）
- 全量 P1 账号（约 84 个）
- 按 `teams` 字段筛选

**GitHub Release 信源（类别 12）**：
- 返回所有 `releases.atom` URL 列表
- 按 `teams` 字段筛选：每个团队只返回与自身业务线相关的仓库
  - wallet 团队 → metamask, walletconnect, account-abstraction, viem, wagmi, safe 等
  - xlayer 团队 → optimism, op-geth, zksync-era, consensus-specs 等
  - pay 团队 → evm-cctp-contracts, EIPs, dss, ethena-contracts 等
  - dev 团队 → openzeppelin-contracts, foundry, hardhat, graph-node, chainlink 等
  - all 标签的仓库每个团队均返回

**RSS/博客信源（类别 13）**：
- 返回所有 RSS URL 列表
- 按 `teams` 字段筛选
- 标注采集方式（浏览器 fetch）

**数据平台信源（类别 14）**：
- 返回平台 URL 和采集方式
- 仅在明确请求或检测到异常时采集（非每次常规运行）

### 模式 B：接收情报候选 & 快速摄入（Intel-Driven Update）

**触发条件**：pipeline 结束后传入"新信息源候选"列表（来自 `web3-intel-filter` 输出的 `🔍 新信息源候选` 区块）。

**快速摄入流程**（5 步）：

1. **读取候选列表**：从传入内容中提取待验证账号（Handle、建议类别、发现场景、建议优先级）

2. **快速验证**（每个账号）：
   - 搜索 `site:x.com/@{handle}` 或 `"{账号名}" twitter X 2026`，确认账号存在
   - 确认近 30 天内有发帖（活跃性）
   - 确认账号内容与描述相符

3. **判断是否收录**：
   - 已存在于 sources.md → 跳过，记录"已收录"
   - 通过验证 → 进入下一步
   - 验证失败（账号不存在/已停更） → 记录"已拒绝"原因

4. **写入 sources.md**：
   - 将通过验证的账号追加到对应类别表格
   - 补充所有字段：Handle、名称、说明、粉丝量级、优先级、teams 标签、最后验证日期（今天）
   - 更新文件头部的"最后全量更新"日期和"本次新增"备注

5. **输出摄入报告**：
   ```markdown
   ## 信源摄入报告 — YYYY-MM-DD（Intel-Driven）

   ### 新增（N个）
   | Handle | 类别 | 优先级 | 说明 |
   |--------|------|--------|------|

   ### 已存在（N个）
   ### 拒绝（N个）
   ### 待人工复核（N个，需更多信息才能判断）
   ```

### 模式 C：手动全量更新

用户主动触发（每周或重大事件后），按以下步骤执行：

1. **读取现有列表**：从 `references/sources.md` 加载当前信息源
2. **搜索新信息源**：
   - 搜索近期热门 Web3 项目、新上线的链、获融资项目
   - 重点关注与 OKX 四条业务线相关的新动态
   - 关键词：`new web3 project 2026 twitter`、`trending crypto developer twitter`
3. **验证所有候选**：同模式 B 的验证步骤
4. **更新列表 & 输出变更摘要**

## 质量标准

每次维护列表时，确保：

1. **Handle 准确性**：每个 @handle 对应正确的账号，不是同名或冒充号
2. **活跃度**：所有收录账号近 30 天内有发帖
3. **去重**：同一个人/项目只收录一次
4. **关联性**：每个账号都有明确的 OKX 业务关联说明
5. **时效性**：`最后验证日期`不应超过 30 天
6. **teams 标签完整**：每个账号都有正确的 teams 标签（`all` / `wallet` / `xlayer` / `pay` / `defi` / `dev` / `growth`）

## 优先级评定标准

- **P0**：直接竞品 / 核心合作方 / 对 OKX 产品决策有直接影响
- **P1**：间接相关的头部项目/人物 / 行业标杆
- **P2**：值得关注但非核心 / 可能成为未来 P1 的观察对象

## 注意事项

- 这个 skill **不负责内容采集**，只负责信息源管理
- 新兴项目（类别 2）和 GitHub 仓库（类别 12）变动最快，更新时优先检查
- 中国/亚洲区的 Web3 builder 对 OKX 尤其重要，注意覆盖
- Intel-Driven Update（模式 B）优先于手动全量更新，频率可以更高
- RSS/数据平台 URL 需定期验证可访问性（每月一次），失效后在 sources.md 标注 `[DEAD]`
- GitHub releases.atom URL 格式固定为 `https://github.com/{owner}/{repo}/releases.atom`，VM 可直接 curl
