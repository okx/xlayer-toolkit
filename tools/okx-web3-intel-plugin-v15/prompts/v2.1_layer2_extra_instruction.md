## severity 判定标准

### CRITICAL（立即响应）

满足以下任一条件：
- 竞品钱包（MetaMask、Phantom、Backpack、Rainbow）发布整代版本更新，包含影响用户关键路径的颠覆性功能
- AI Agent 链上标准（ERC-8004、ERC-8183、x402 等）被主流链强制采纳或首次生产环境落地
- OKX Wallet 直接接入的协议发生安全漏洞（无论金额）
- 跨链桥安全事件（不设金额阈值）

### HIGH（尽快处理）

满足以下任一条件：
- 竞品钱包发布影响用户关键路径的重大功能（非整代更新，但改变核心体验）
- AA / ERC-4337 / ERC-7702 标准有重大进展（Paymaster、SessionKey、Bundler 落地）
- 主流链的 Agent SDK / Framework 测试网或主网上线
- AI Agent 与 DeFi 协议的原生集成（无人干预链上执行）
- 主流公司/协议明确 AI Agent 基础设施方向的战略表态（如 Coinbase、Circle 等）
- 主流 DApp（Uniswap、OpenSea 等）切换钱包集成标准
- 底层协议重大升级影响产品架构或用户体验（以太坊硬分叉、Solana 重大客户端升级等）
- 新技术原语从"概念/研究"进入"工程落地"节点（主网部署、标准化提案进入 Final/Last Call）
- 监管执行层面动作直接改变 OKX 产品能做什么/不能做什么
- 机构 DeFi / RWA 新协议或新产品上线，直接影响 OKX Wallet DeFi 入口
- 社交登录、passkey、无助记词方案在竞品落地（Web2 用户进入门槛变化）
- DeFi 协议攻击损失 > $1M

### MEDIUM（排期跟进）

满足以下任一条件：
- 行业趋势或协议里程碑，与 Wallet / AI Agent 业务线有间接关联
- 竞品灰度/内测阶段的重大功能（方向已明确但未全量）
- 跨链基础设施升级（Polygon、ZKsync 等影响 Wallet 多链支持的新功能）
- 开发工具链或基础设施演进影响工程效率或技术选型
- 开发者生态资源流向有数据支撑的变化
- 技术仍在研究阶段但已有头部团队投入工程资源（标注「待观察」）
- Agent 钱包/账户抽象与 AI 结合的新方案（概念阶段，尚无生产落地）

### LOW（了解即可）

满足以下任一条件：
- 各链硬分叉/升级影响 RPC 兼容性，但无需产品层改动
- 牌照制度变化（非直接影响 OKX 的市场准入路径）
- 远期升级规划、行业趋势报告
- 监管讨论稿或征求意见稿（非执行层面动作）

### INFO（与本业务线无关 → 强制降级）

以下事件类型无论内容多重要，一律强制 INFO：

- Crypto Card 相关内容（竞品发卡、Card 新功能、Card 返利、Card 接入新链）
- 纯 AI 模型/算法进展（无链上交互，不属于 Web3 范畴）
- AI 项目融资（无产品落地细节）
- L2 技术架构细节（XLayer 团队职责范围，不在本 Profile 范围内）

## 分析要点

### §1 竞品钱包产品动作分析

当事件涉及竞品钱包（MetaMask、Phantom、Coinbase Wallet、Rainbow、Trust Wallet、Backpack）时：

在 impactSummary 中必须回答：
- 这个功能影响的是用户关键路径中的哪个环节？（创建钱包、Swap、跨链、DApp 连接、Agent 交互）
- OKX Wallet 当前是否有对标能力？差距有多大？
- 该功能是 UI 级迭代还是产品方向性变化？（UI 迭代 → 降级为 INFO）

在 productCompetitiveness 维度中评估：
- 用户感知差距：普通用户能否在 1 分钟内感知到差异
- 技术壁垒：OKX Wallet 跟进该功能的工程复杂度（高/中/低）
- 窗口期：竞品领先后，OKX 有多少时间跟进而不丢失用户

典型判断参考：
- MetaMask 集成 RWA 代币化股票 → HIGH（开辟新业务线）
- Phantom 推出钱包内消息 → HIGH（影响用户路径）
- Base 脱离 OP Stack → HIGH（战略级变动，影响 Superchain 生态）
- MetaMask 更新 UI 颜色 → INFO（UI 迭代，非战略级）
- Coinbase Wallet 优化动画效果 → INFO
- Bybit 上线新交易对 → INFO（CeFi 功能）

### §2 AA / 智能账户标准分析

当事件涉及 ERC-4337、ERC-7702、ERC-7579、Paymaster、SessionKey、Bundler、模块化账户时：

在 protocolCompatibility 维度中评估：
- OKX Wallet Smart Accounts 当前实现是否兼容该标准变更
- 是否需要 SDK 层或签名流程层的适配
- 该标准进入 Final/Last Call 的时间线

在 impactSummary 中必须说明：
- 该标准成熟后，用户与钱包的交互方式会不会改变
- 竞品（MetaMask、Safe）在该标准上的跟进状态

### §3 AI Agent 链上基础设施分析

当事件涉及 AI Agent 链上标准（ERC-8004、ERC-8183、x402/MPP、Agent SDK/Framework、Agent+DeFi 集成）时：

在 agentInfrastructure 维度中评估：
- 该标准/SDK 是否进入生产可用阶段（主网部署 vs 测试网 vs 概念阶段）
- 是否有头部项目（Coinbase AgentKit、Polygon Agent CLI、Uniswap MPP Skill 等）已宣布采用
- OKX Wallet Smart Accounts 的 Agent 接入路径是否受影响

在 impactSummary 中必须回答：
- Agent 生态的开发者会围绕哪个钱包的权限模型构建？OKX 是否有被边缘化的风险？
- 该基础设施落地后，Agent 支付/交易场景的默认路由是谁？

当前最高关注度话题：AI Agent 链上基础设施标准落地。标准落地、SDK 上线、首笔实际执行均为高价值信号，优先提升 severity。

### §4 底层协议升级分析

当事件涉及以太坊硬分叉、Solana 重大客户端升级、各链硬分叉影响 RPC 兼容性时：

在 protocolCompatibility 维度中评估：
- 升级是否影响 OKX Wallet 的签名流程、Gas 计算、交易构建
- 是否需要 RPC 节点版本升级
- 升级窗口多远（<3 天 = HIGH，1-2 周 = MEDIUM，>1 月 = LOW）

### §5 新技术原语分析

当事件涉及新技术首次生产落地（zkEVM 证明时间突破、意图交易框架集成、零知识隐私原语等）时：

核心判断问题："该技术成熟后，用户与产品的交互方式会不会改变？"

进入工程落地阶段的标志（提升 severity）：
- 主网部署（非测试网）
- 标准化提案进入 Final/Last Call
- 头部项目宣布采用该技术方案

仍在研究阶段但有头部团队投入资源 → MEDIUM 并在 impactSummary 中标注「待观察」

### §6 监管与合规分析

当事件涉及监管/合规时：

严格区分执行层面动作与讨论层面：
- 执行层面（正式规则生效、牌照新规生效、SEC 正式启动规则制定）→ 按影响评级
- 讨论层面（"某国监管机构正在考虑"、行业协会建议书、国会听证讨论）→ INFO

在 compliance 维度中评估（仅当直接影响 OKX 产品路径时）：
- 该监管动作是否改变 OKX Wallet / AI Agent 产品的能力边界
- 是否影响 OKX 的市场准入路径

### §7 机构 DeFi / RWA 分析

当事件涉及 RWA 代币化、代币化证券、代币化存款时：

区分保留和丢弃：
- KEEP（提升 severity）：新协议/新产品上线、直接影响 OKX Wallet DeFi 入口的 RWA 集成事件
- DROP（INFO）：纯 TVL 数字里程碑（如"RWA TVL 破 $X 亿 ATH"）、RWA 项目融资公告

在 ecosystemOpportunity 维度中评估：
- 该 RWA 产品是否可集成到 OKX Wallet 的 DeFi/Earn 入口
- 代币化证券是否影响 OKX Wallet 内的资产展示方式（如需区分 7×24 交易资产与受交易时段限制的资产）

### §8 跨链与多链基础设施分析

当事件涉及跨链桥、跨链消息协议、多链基础设施升级时：

在 ecosystemOpportunity 维度中评估：
- 该基础设施变化是否影响 OKX Wallet 的跨链 Swap 路由
- 新的跨链标准是否需要 OKX Wallet 适配

安全事件特殊规则：跨链桥安全事件不设金额阈值，一律至少 HIGH，标注是否影响 OKX Wallet 接入的协议。

### §9 GitHub Release 特殊规则

当 source_type 为 GitHub Release 类信息时：
- P0 仓库的 major/minor 版本发布（v1.x.0, v2.x.0）→ 至少 MEDIUM，不需要通过关键词评分
- patch 版本（v1.x.y）仅当 changelog 含 security fix / breaking change 时提升 severity
- 纯 chore/docs 的 pre-release tag（rc.1, alpha.x）→ INFO，除非包含核心功能
